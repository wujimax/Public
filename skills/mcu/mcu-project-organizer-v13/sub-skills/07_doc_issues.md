## 执行前：判断运行模式

### 模式判断
1. 检查目标文档是否存在（`07_已知问题与建议.md`）
2. 存在 → **补厚模式**：读取现有文档，只补充缺失内容
3. 不存在 → **全新生成模式**：从零开始生成

### 补厚模式操作规则
- 读取现有文档，记录"已有哪些章节/内容"
- 与本子skill的"应有内容"对比，找出缺失项
- 只追加缺失内容，不修改已有内容
- 在追加内容前加注释：`<!-- 补充于 [日期] -->`（可选）
- 执行完成后输出操作摘要

### 数据读取规则（全新/补厚均适用）
1. 先读 `_analysis/issues_and_risks.md` → 了解有哪些源文件需要读
2. 直接打开 `_v10_snapshot/sources/` 中的源文件
3. 从源码提取完整数据（数值必须有 源文件:行号 来源）
4. `_analysis/` 只作为导航，文档中所有数据来自源码

## 文档结构

### 必须按严重程度分级

#### P0 严重BUG

代码逻辑错误（数组越界、条件判断错误、NVIC冲突等）

每个P0问题必须包含:
- 文件名:行号
- 问题现象
- 影响范围
- **代码片段展示错误所在**
- 建议修复方式

**示例**:
```markdown
##### P0-001: 数组越界风险
- **文件**: app_sensor.c:125
- **现象**: `sensor_data[MAX_SENSOR]` 访问超出数组边界
- **代码**:
  ```c
  for(int i = 0; i <= MAX_SENSOR; i++) {  // 应为 < 而非 <=
      sensor_data[i] = 0;
  }
  ```
- **影响**: 覆盖相邻内存变量，导致不可预测的行为
- **修复建议**: 将 `<=` 改为 `<`
```

#### P1 注释错误

注释与代码不一致。

每个P1问题必须包含:
- 文件名:行号
- 注释内容
- 实际代码
- 正确描述

#### P2 设计风险

安全漏洞、可靠性风险、资源竞争。

每个P2问题必须包含:
- 风险描述
- 产生条件
- 影响程度
- 缓解建议

#### P3 改进建议

性能优化、维护性改进、编码规范。

每个P3建议必须包含:
- 建议内容
- 预期收益
- 实施难度

## BUG验证要求

对每个 P0 BUG，如果 `_analysis/issues_and_risks.md` 中的描述不够详细:
1. 回到 `_v10_snapshot/sources/` 读取对应行号
2. 确认代码确实存在问题（不是AI误判）
3. 提取代码片段作为证据

**禁止虚构BUG。如果不确定是否是BUG，降级为 P2 设计风险并注明 "[待确认]"。**
