> **职责**: 项目代码的"人类可读版本"，让工程师不看源码也能知道每个函数干什么。
> **输入**: `_v10_snapshot/sources/`, `_v10_context/call_graph_hint.json`, `_analysis/module_analysis.md`
> **输出**: `10_代码语义化.md`

## 执行前：判断运行模式

### 模式判断
1. 检查目标文档是否存在（`10_代码语义化.md`）
2. 存在 → **补厚模式**：读取现有文档，只补充缺失内容
3. 不存在 → **全新生成模式**：从零开始生成

### 补厚模式操作规则
- 读取现有文档，找出已有函数卡片的列表
- 对照 `call_graph_hint.json` 的函数清单，找出缺失的函数
- 只补充缺失的函数卡片，已有的不动
- 在文档末尾追加新章节 `## 补充函数卡片`
- 执行完成后输出操作摘要

### 数据读取规则（全新/补厚均适用）
1. 先读 `_analysis/module_analysis.md` → 了解有哪些源文件和函数
2. 读 `_v10_context/call_graph_hint.json` → 获取完整函数定义列表
3. 直接打开 `_v10_snapshot/sources/` 中的源文件，读取每个函数的实现
4. `_analysis/` 只作为导航，函数卡片中所有数据来自源码

## 处理优先级

```
优先级1（必须处理）：
- 包含 main() 的文件
- 业务逻辑主文件（app.c 等）
- 调度器文件（OS_System.c）
- 被最多其他文件 include 的头文件

优先级2（尽量处理）：
- 所有 hal_*.c 驱动文件
- 通信协议文件（hal_NBIot.c, hal_rfd.c 等）
- 参数管理文件（para.c 等）

优先级3（文件头+主要函数，不做完整卡片）：
- 工具类函数文件
- 第三方库：只记录"这个库是干什么的"，不分析函数
```

**返回值**：无（连接结果通过回调 `BC26_ConnectCallback` 通知）

**置信度**：⚠️ AI推断（函数无注释，根据AT指令序列逻辑推断）

**代码位置**：`hal_NBIot.c:241`

## 置信度判断规则

| 情况 | 标注 |
|------|------|
| 函数有完整注释，内容来自注释 | `✅ 直接提取` |
| 函数有部分注释，AI补充了推断 | `⚠️ AI推断（基于注释+代码）` |
| 函数无注释，逻辑清晰（<50行）| `⚠️ AI推断` |
| 函数无注释，逻辑复杂（>50行）| `⚠️⚠️ 不确定 — 建议查阅原始代码` |

---

## 生成卡片时的AI自检规则

1. 函数名在源码中真实存在（从 `call_graph_hint.json` 或源码确认）
2. 参数和返回值从函数签名直接提取，不能用推测
3. "在哪里被调用"必须从 `call_graph_hint.json` 的 `function_calls` 获取并在源码中验证
4. "它会调用"同样必须从调用图或源码确认
5. "代码位置"必须是真实行号
6. 功能描述不超过30字，动词开头

---

## 输出要求

- 优先级1的文件必须100%覆盖所有公开函数
- 优先级2的文件尽量覆盖所有公开函数
- 优先级3的文件只需文件头摘要
- 每张卡片必须有"代码位置"字段
- 复杂函数（>50行且无注释）必须有⚠️⚠️标注
