# 子Skill 14: 生成 / 追加 12_历史经验库.md

> **职责**: 将人类真实经验（聊天记录、Word文档、邮件、口述笔记、历史FAQ）结构化写入知识库，补充代码分析无法覆盖的"经验层"知识。
> **输入**: 用户提供的原始文本（见"输入格式规范"）+ 已有知识库文档（用于关联和去重）
> **输出**: `12_历史经验库.md`

---

## 与其他子skill的本质区别

| 对比项 | 代码类子skill（01-13） | 本子skill（14） |
|-------|----------------------|----------------|
| 事实来源 | `_v10_snapshot/` 源码 | 用户提供的文本 |
| 铁律8适用性 | 必须遵守 | **不适用**（无源码可读） |
| 事实可信度 | 代码即事实 | 按来源类型分级标注 |
| 错误风险 | AI推断错误 | 信息来源本身可能有误 |
| 核心约束 | 不得凭空推断 | 不得篡改原意，必须标注可信度 |

**铁律8对本子skill不适用。** 本子skill的"事实"是人类提供的文本，不是源码。

---

## 必须遵守的规则

开始前先读取:
- `shared/iron_rules.md` — 铁律 9（补厚不覆盖）**对本子skill适用**

本子skill专属规则（优先级高于iron_rules中的代码类规则）：

**规则E1：忠实原意，不过度提炼**
不得在提炼过程中改变原始描述的含义。如果原文模糊，保留模糊，不要替用户"补全"原因。

**规则E2：可信度必须标注**
每条经验条目必须根据来源类型标注可信度，不得省略。

**规则E3：相关性必须判断**
每条输入内容必须判断是否与当前项目相关。不相关的内容不写入知识库，在执行摘要中说明原因。

**规则E4：不与11_FAQ重复**
写入前必须检查 `11_常见问题清单.md`，如果同一个问题已有高质量答案，不重复写入，改为在FAQ对应条目后追加"经验补充"标注。

**规则E5：关联代码尽力但不强制**
尽量将经验条目关联到具体代码位置（文件:行号），但如果知识库中找不到对应位置，可以留空，不得伪造行号。

## 执行前：判断运行模式

### 模式判断
1. 检查 `12_历史经验库.md` 是否存在
2. 存在 → **追加模式**：读取现有文档，在末尾追加新条目，不修改已有内容
3. 不存在 → **全新生成模式**：创建文档并写入第一批条目

### 追加模式操作规则
- 读取现有文档，记录当前最大序号（如已有 E08，则新条目从 E09 开始）
- 执行去重检查（见"去重流程"）
- 只在文档末尾追加，不修改已有条目
- 执行完成后输出操作摘要

## 输入格式规范

用户提供内容时，必须使用以下格式（每条经验一个块）：

```
【来源】微信群 / 邮件 / 口述 / Word文档 / 项目记录 / 其他
【时间】2024-03（可选，不知道可不填）
【内容】
（粘贴原文、摘要或口述记录，不限格式）
```

**允许一次性提供多个块**，每个块之间用空行或分隔线隔开即可。

**如果用户没有按格式提供**：
- 不要拒绝执行
- 尽力从上下文推断来源类型，在条目中标注"[来源未标注]"
- 可信度自动降一级

---

## 处理流程

### 第1步：拆分与识别

对每个输入块，识别其中包含的"经验单元"数量。一段文字可能包含多个独立问题：

```
判断标准：一个经验单元 = 一个独立的"问题/现象 + 解法/结论"对
如果一段文字包含3个不同问题的解法，拆分为3条独立条目
```

### 第2步：相关性判断

对每个经验单元，判断是否与当前项目相关：

**读取以下文档辅助判断**：
- `01_项目介绍.md`（项目功能边界）
- `02_硬件配置.md`（外设和芯片型号）
- `05_通信协议.md`（协议类型）

**判断逻辑**：
- 提到的芯片型号/外设/协议与当前项目一致 → 相关
- 描述的场景与项目功能重叠 → 相关
- 通用嵌入式经验（非项目特定）→ 标注"[通用经验]"，仍写入，但降低可信度权重
- 明确属于其他项目/芯片/场景，无迁移价值 → 不写入，在执行摘要中说明

### 第3步：去重检查

**读取** `11_常见问题清单.md`（如果存在）和现有 `12_历史经验库.md`：

| 情况 | 处理方式 |
|------|---------|
| 完全相同的问题，FAQ已有高质量答案 | 不新建条目；在FAQ对应条目末尾追加 `> 💬 经验印证：[简短说明]` |
| 相似问题，角度不同 | 新建条目，在"关联条目"字段引用FAQ编号 |
| 全新问题，FAQ未覆盖 | 直接新建条目 |

### 第4步：结构化提炼

对每个通过前三步的经验单元，提炼为标准格式（见下文）。

**提炼原则**：
- 现象：**原文描述**，不改写，可精简但不改意
- 原因：如果原文有明确原因，照录；如果是AI推断，必须标注 `[AI推断]`
- 解法：步骤化，如果原文只有结论没有步骤，原样写入
- **禁止**：把"有时候会出问题"改写成"必然触发XXX"这类加强结论

### 第5步：关联代码（尽力）

对每个经验条目，尝试在以下文档中找到对应代码位置：
- `06_关键参数表.md`（参数类问题）
- `05_通信协议.md`（协议类问题）
- `07_已知问题与建议.md`（BUG类问题）
- `04_功能模块.md`（功能类问题）

找到则填写"关联代码"字段；找不到则留空，**不伪造行号**。

### 第6步：分类与写入

按类别归入对应章节，分配递增序号（E01, E02...），写入文档。

## 可信度评级标准

| 等级 | 标志 | 适用场景 |
|------|------|---------|
| 高可信 | ★★★ | 当事人亲历并记录；有截图/日志佐证；多人独立印证 |
| 中可信 | ★★ | 转述他人经历；口述但有细节；来自项目记录文档 |
| 低可信 | ★ | 口述无细节；来源不明；AI根据上下文推断原因 |

**可信度降级规则**：
- 来源未标注 → 自动降一级
- 原因是AI推断 → 该字段单独标注 `[AI推断]`，整体可信度不降级
- 通用经验（非项目特定）→ 可信度不影响，但在条目末尾加 `[通用经验，需在本项目验证]`

## 实际示例

```markdown
---
**E01** [调试类] NBIoT 模块上电后偶发性初始化失败

**来源**：微信群聊 | **时间**：2024-03 | **可信度**：★★★

**现象**：
上电后 NBIoT 模块有时候能正常初始化，有时候一直卡在等待 AT 响应。重启主控后恢复。
复现概率约 20%，低温环境下更容易触发。

**原因**：
模块上电后需要约 3 秒稳定时间，但代码里等待时间只有 1.5 秒，低温下模块启动更慢。

**解法**：
1. 将初始化等待时间从 1500ms 改为 4000ms
2. 增加 AT 指令重试机制（最多重试 3 次，间隔 1 秒）
3. 如果改完仍有问题，检查模块供电是否稳定（用示波器看上电瞬间的电压跌落）

**关联代码**：
- `hal_NBIot.c:241` — 初始化流程入口，等待时间在此处配置

**关联条目**：Q09（NBIoT连接不上的排查流程）

---

**E02** [接手类] 项目有两套配置文件，新人容易搞混

**来源**：口述（前任工程师交接） | **时间**：未知 | **可信度**：★★

**现象**：
`config.h` 和 `para.c` 里都有参数定义，但职责不同。之前有工程师改了 `config.h` 发现不生效，
原来运行时参数从 EEPROM 读取覆盖了编译时的默认值。

**原因**：
`config.h` 是编译期默认值，`para.c` 负责从 EEPROM 读取运行时参数并覆盖默认值。
首次烧录时 EEPROM 为空，用默认值；之后每次上电都以 EEPROM 存储的值为准。

**解法**：
修改运行时参数用上位机或调试串口指令写入 EEPROM，不要直接改 `config.h`（改了烧录也不生效）。
如果需要恢复出厂默认，长按复位键 5 秒（见 `app.c` 中的恢复出厂逻辑）。

**关联代码**：
- `para.c:1` — 参数读写逻辑
- `config.h:1` — 编译期默认值定义

**关联条目**：Q07（如何修改运行时配置参数）

---
```
