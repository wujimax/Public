# MCU Project Organizer V14 通用化优化方案（合并版）

> 生成日期：2026-04-20
> 来源：Claude 三层架构方案 + GPT 证据分级方案合并
> 背景：将 V13（仅支持 Keil/IAR）升级为通用嵌入式项目知识库构建工具，覆盖 Betaflight、Grbl、LoRaMac-node、QMK、VESC、OpenMV、DAPLink 等主流开源项目。

---

## 根本问题诊断（三层同时失效）

| 层次 | 问题 | 影响 |
|------|------|------|
| **提取层** (bootstrap) | 只认 `.uvprojx`/`.ewp`，其他全部报错 | Betaflight/Grbl/LoRaMac 等完全无法启动 |
| **分析层** (00_analysis_engine) | 不知道哪些宏激活，把 `#ifdef` 的死代码当活代码分析 | 文档包含未编译代码的误导信息 |
| **文档层** (iron_rules + 子skill) | 假设所有项目都有 GPIO_Init、NVIC、Keil 外设配置 | 对 QMK 键盘固件、Grbl CNC 固件生成方向错误的文档 |

> **关键认知**：只修 bootstrap 不够，后面文档还是会生成"格式齐全、内容走偏"的结果。三层必须同步升级。

## 核心设计原则（GPT 补充）

### 原则 A：证据分级（贯穿所有层）

提取结果必须携带置信度标签，后续分析层和文档层根据置信度决定断言强度：

| 级别 | 名称 | 来源 | 断言权限 |
|------|------|------|---------|
| L1 | `compile_database` | 真实 `compile_commands.json` | 可断言"活代码"/"死代码" |
| L2 | `build_inferred` | 编译器预处理 (`gcc -E -dM`) 成功执行 | 可断言，需标注来源 |
| L3 | `heuristic_scan` | 静态解析 Makefile/CMake | 只能辅助，不得断言 |
| L4 | `manual` | 用户手动指定 | 取决于用户输入完整度 |

### 原则 B：条件编译过滤不是编译器（GPT 核心警告）

以下情形 PowerShell/Python 静态解析**无法可靠处理**：

```c
// 复杂条件，静态解析极易出错
#if defined(USE_BARO) && (BARO_TYPE == BARO_BMP280)
#if !defined(CONFIG_FOO) || CONFIG_LEVEL > 2
#if HW_VERSION >= 60
```

还会遇到：宏值是数字/枚举而非布尔、`#undef`/重定义、多层 include 展开、CMake generator expression、Kconfig 生成的 `.config`、Makefile target 动态拼接 CFLAGS。

**结论**：`active_code_regions.json` 必须携带 `confidence` 字段；低置信结果只能辅助 AI，不得作为"死代码"的最终判决。

### 原则 C：提取优先级（GPT 补充）

```
1. 先找已有 compile_commands.json          → L1，直接使用
2. 有条件时生成 compile_commands.json       → L1，调用 cmake --build 或 bear make
3. 再用构建系统 adapter 静态推导            → L3
4. 最后才 heuristic/manual               → L3/L4
```

## 三层优化方案

---

### Layer 1：通用提取层 (bootstrap 重构)

**核心设计：构建系统适配器模式**

```
bootstrap_v14.cmd
  └─→ 00_detect_build_system.ps1      # 探针：识别项目类型
  └─→ 01_find_compile_commands.ps1    # 新增：优先查找/生成 compile_commands.json
  └─→ 02_select_target.ps1            # 交互：用户选择编译目标
  └─→ 03_extract_context.ps1          # 路由：按构建类型调用对应适配器
       ├─ adapter_keil.ps1            # 保留 V13 逻辑
       ├─ adapter_iar.ps1             # 保留 V13 逻辑
       ├─ adapter_makefile.ps1        # 新增：解析 Makefile（带 confidence）
       ├─ adapter_cmake.ps1           # 新增：解析 CMakeLists.txt（带 confidence）
       ├─ adapter_yaml.ps1            # 新增：DAPLink 式 YAML 项目
       └─ adapter_manual.ps1          # 兜底：用户手动指定（带 confidence=L4）
  └─→ 04_filter_active_code.ps1      # 新增：条件编译过滤（带 confidence）
  └─→ 05_export_snapshot.ps1          # 原 02：导出快照（保留）
  └─→ 06_build_call_graph.ps1         # 原 03：调用图（保留）
```

**各适配器的提取策略：**

| 构建类型 | 检测特征 | 宏提取方式 | 源文件提取方式 | 置信度 |
|----------|----------|-----------|---------------|--------|
| compile_commands | `compile_commands.json` | 解析每条编译命令的 `-D` | 每条命令的 `file` 字段 | L1 |
| Keil | `*.uvprojx` | XML `<Define>` 节点 | XML `<FilePath>` 列表 | L2 |
| IAR | `*.ewp` | XML `CCDefines` | XML `<file>` 列表 | L2 |
| Makefile | `Makefile` + `CC =` | `DEFS`/`CFLAGS` 中的 `-DXXX` | `SOURCES`/`SRC`/`OBJS` 变量 | L3 |
| CMake | `CMakeLists.txt` | `add_definitions`/`target_compile_definitions` | `add_executable`/`target_sources` | L3 |
| DAPLink YAML | `projects.yaml` | 解析 yaml → 查生成的 uvproj | 同上 | L3 |
| 手动兜底 | 任何 | 用户输入宏列表 | 用户指定扫描目录 | L4 |

**提取输出结构（GPT 补充，所有适配器必须遵守）：**

```json
// _v10_context/project_context.json
{
  "project_name": "Betaflight",
  "target_name": "BETAFPVF405",
  "extraction_mode": "heuristic_scan",
  "confidence": "L3",
  "warnings": [
    "Makefile 中 CFLAGS 含 shell 展开，部分宏可能未被静态解析捕获",
    "target 逻辑分布在 mk/*.mk，已尝试递归解析，完整性无法保证"
  ],
  "active_macros": [...],
  "active_sources": [...],
  "candidate_sources": [...],
  "excluded_sources": [...],
  "target_evidence": "src/target/BETAFPVF405/target.h 存在，内含 TARGET 宏定义"
}
```

| 字段 | 含义 |
|------|------|
| `active_sources` | 高置信确认参与编译的源文件 |
| `candidate_sources` | 可能参与编译但未确认（条件编译等） |
| `excluded_sources` | 确认不参与编译的源文件 |
| `target_evidence` | 判断当前 target 的依据（引用具体文件/行） |
| `warnings` | 静态解析的局限说明 |

**`04_filter_active_code.ps1`（条件编译过滤器）：**

功能：给定 `active_macros`，扫描源文件识别 `#ifdef`/`#if defined()`/`#ifndef` 块，判断活代码/死代码。

输出 `active_code_regions.json`（每条必须包含 `confidence`）：

```json
{
  "src/drivers/barometer.c": {
    "always_active": [[1,45], [200,350]],
    "conditionally_active": [
      {
        "range": [46,199],
        "condition": "USE_BARO_BMP280",
        "active": true,
        "confidence": "high"
      },
      {
        "range": [351,500],
        "condition": "USE_BARO_MS5611",
        "active": false,
        "confidence": "high"
      },
      {
        "range": [501,620],
        "condition": "CONFIG_LEVEL > 2",
        "active": "unknown",
        "confidence": "low",
        "reason": "数值比较条件，静态解析无法判断"
      }
    ]
  }
}
```

`confidence` 取值：
- `high`：简单布尔宏，`active_macros` 中有/无该宏，可直接判断
- `medium`：宏值已知但条件涉及运算，可推导
- `low`：复杂条件、宏值未知、多层展开，无法判断
- `unknown`：保留原始条件表达式，交给 AI 自行判断

### Layer 2：分析引擎增强

**`00_analysis_engine.md` 三处关键修改：**

#### 修改1：首先读取 extraction_mode/confidence/warnings（GPT 补充）

分析引擎执行的**第一步**（优先于读任何源文件）：

```markdown
### 前置：提取质量检查
1. 读取 project_context.json 中的 extraction_mode、confidence、warnings
2. 根据置信度确定分析策略：
   - L1/L2：可信任 active_sources，直接分析
   - L3：active_sources 仅供参考，同时检查 candidate_sources
   - L4：手动模式，提示用户核查完整性
3. warnings 中的每一条，在分析报告中对应章节标注 ⚠️
```

#### 修改2：条件编译死代码排除（铁律 8.1，带置信度约束）

> **铁律 8.1 - 条件编译死代码排除**
>
> - `confidence: high` 的 `active: false` 区域：禁止提取任何内容写入文档
> - `confidence: medium` 的 `active: false` 区域：不写入主文档，在"已知问题"中标注 `[可选编译路径]`
> - `confidence: low/unknown` 的区域：**不得作为死代码处理**，必须读取并分析，在文档中标注 `[条件编译，当前 target 激活状态不确定]`
>
> 严禁将低置信过滤结果当作最终判决。

#### 修改3：输出 project_profile.json

```json
// _analysis/project_profile.json
{
  "project_type": "flight_controller",
  "target_chip": "STM32F405RGT6",
  "target_name": "BETAFPVF405",
  "extraction_confidence": "L3",
  "has_rtos": false,
  "has_networking": false,
  "has_filesystem": false,
  "primary_protocol": "UART_ESC_DSHOT",
  "domain_keywords": ["PID", "motor", "gyro", "OSD"]
}
```

支持的 `project_type` 枚举值：

| 值 | 适用项目 | 核心文档关注点 |
|----|---------|--------------|
| `flight_controller` | Betaflight、iNav | PID、陀螺仪、电调协议、OSD |
| `cnc` | Grbl、Marlin | G-code 解析、运动规划、步进驱动 |
| `keyboard` | QMK、ZMK | 键盘矩阵、keymap、USB HID、layer |
| `iot_lorawan` | LoRaMac-node | MAC 状态机、Region、Radio driver |
| `motor_controller` | VESC | FOC、CAN 协议、hwconf |
| `debug_probe` | DAPLink | DAP 协议、目标侧 Flash 算法 |
| `machine_vision` | OpenMV | 图像 pipeline、传感器 I2C |
| `generic_mcu` | Keil/IAR 私有项目 | 通用 MCU 外设配置 |

**分析任务自适应（不适用任务标注 N/A，不强行生成）：**

| 分析任务 | flight_ctrl | cnc | keyboard | iot_lorawan | generic_mcu |
|---------|-------------|-----|----------|-------------|-------------|
| NVIC优先级 | ✅ | ✅ | ✅ | ✅ | ✅ |
| GPIO引脚表 | ✅ | ✅ | ⚠️ 仅矩阵 | ✅ | ✅ |
| 通信协议 | ✅ DSHOT/SBUS | ✅ G-code | 改为 USB HID | ✅ LoRa/AT | ✅ |
| 内存布局 | ✅ | ✅ | ✅ | ✅ | ✅ |
| 时钟树 | ✅ | ✅ | ✅ | ✅ | ✅ |

### Layer 3：文档层自适应

**新增铁律 11：项目类型自适应**

> 读取 `_analysis/project_profile.json` 中的 `project_type`，对不适用的文档章节标注 `[N/A for {type}: {原因}]` 而非强行生成。
>
> 同时读取 `extraction_confidence`：
> - L3/L4 置信度时，文档开头必须增加 ⚠️ 提示：
>   `> ⚠️ 本知识库基于 {heuristic_scan/manual} 提取，{N} 条 warnings 待确认，请结合实际构建环境验证关键参数。`

**各子 skill 新增"类型自适应"入口（每个文档子 skill 开头执行）：**

```markdown
### 前置：类型与置信度检查
1. 读取 _analysis/project_profile.json
2. 根据 project_type 确定章节模板
3. 根据 extraction_confidence 确定文档置信度标注
4. 对不适用章节写：> [本章节对 {project_type} 不适用，原因：{reason}]
```

## 设计决策备忘

> **Claude 方案做骨架，GPT 方案做安全边界。**

- 三层架构（提取/分析/文档）和 adapter 模式来自 Claude 方案
- 证据分级（L1-L4）、active/candidate/excluded 分离、条件编译置信度约束来自 GPT 方案
- `04_filter_active_code.ps1` 第一版定位是"启发式辅助"，不是"编译器替代品"
- Makefile/CMake 静态解析只能达到 L3，不能声称等价于真实构建
- 只有 `compile_commands.json`（L1）或编译器预处理结果（L2）才能高置信断言死/活代码
