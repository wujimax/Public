# 子Skill 15: 生成 验收包.md

> **职责**: 从已生成的知识库文档中提取可验证事实，生成结构化的客户验收包。
> 客户通过验收包完成知识库质量验证，不需要读全部文档。
>
> **输入**: 已生成的知识库文档（至少完成 01/02/03/05/06/07/11）+ `shared/anchor-questions.md`
> **输出**: `验收包.md`
>
> **调用时机**: 知识库全量生成完成、09_verification 通过后，交付给客户前。

---

## 必须遵守的共享规则

开始前先读取:
- `shared/iron_rules.md` — 铁律 9（补厚不覆盖）
- `shared/anchor-questions.md` — 20道锚点问题（用于第一部分）

## 验收包结构（四个部分）

```
验收包.md
├── Part 1：锚点自测（20题 + AI自答 + 证据链）
├── Part 2：核心事实清单（40-60条，客户逐条确认）
├── Part 3：关键流程复核（5个流程，客户对照确认）
└── Part 4：盲测问题集（20题，客户自问自答打分）
```

## Part 1：锚点自测

### 执行方法

读取 `shared/anchor-questions.md` 中的 20 道锚点题，逐题从知识库文档中提取答案。

**每题输出格式**：

```markdown
**{题号}** {问题文本}

**知识库答案**：{直接给出答案，结论先行}

**证据**：`{文件名}:{行号}` — {一句话说明代码内容}

**可信度**：{高 / 中 / 低}
{低可信度时加：⚠️ 此答案为AI推断，建议客户重点核实}
```

### 答案提取规则

- 答案必须来自已生成的知识库文档，不得直接读源码补充（验收阶段不重新分析）
- 如果知识库文档里找不到答案 → 写"**⚠️ 知识库未覆盖此问题**"，不要猜
- 证据的文件:行号必须来自知识库文档中记录的引用，不得伪造

### Part 1 末尾输出得分卡

```markdown
## 锚点自测得分卡

| 类别 | 题数 | 有答案 | 无覆盖 | 低可信 |
|------|------|-------|-------|-------|
| 基本信息（A01-A05）| 5 | X | X | X |
| 架构与启动（A06-A10）| 5 | X | X | X |
| 硬件外设（A11-A15）| 5 | X | X | X |
| 通信协议（A16-A18）| 3 | X | X | X |
| 风险参数（A19-A20）| 2 | X | X | X |
| **合计** | **20** | **X** | **X** | **X** |

**初步评级**：{✅合格 / ⚠️局部缺失 / ❌不达标}（按 anchor-questions.md 评分标准）

**建议**：{针对"无覆盖"和"低可信"题目的具体改进建议}
```

## Part 2：核心事实清单

### 执行方法

从以下文档中逐条提取可验证事实，目标 40-60 条：

| 来源文档 | 提取内容 | 目标条数 |
|---------|---------|---------|
| `02_硬件配置.md` | GPIO分配表中每个引脚 | 10-15条 |
| `06_关键参数表.md` | 每个关键参数的数值 | 10-15条 |
| `05_通信协议.md` | 协议关键字段、命令字数值 | 5-10条 |
| `03_系统架构.md` | 初始化顺序、任务列表 | 5-8条 |
| `07_已知问题.md` | P0/P1级别问题 | 3-5条 |

### 每条格式

```markdown
| 编号 | 事实描述 | 来源 | 客户确认 |
|------|---------|------|---------|
| F01 | 芯片型号：STM32F103C8T6 | `01_项目介绍.md` | ☐ ✅正确 ☐ ❌错误 ☐ ❓不确定 |
| F02 | LED1 连接 PC14（GPIO_Pin_14），低电平点亮 | `02_硬件配置.md` | ☐ ✅正确 ☐ ❌错误 ☐ ❓不确定 |
| F03 | 串口1波特率：115200，用于调试输出 | `06_关键参数表.md` | ☐ ✅正确 ☐ ❌错误 ☐ ❓不确定 |
...
```

### Part 2 末尾说明

```markdown
---
**客户操作说明**：
请逐条核对以上事实，在"客户确认"列打勾。
- ✅正确：与您的认知一致
- ❌错误：请在旁边备注正确值
- ❓不确定：需要查源码或询问原开发者后再确认

**重点核查项**：优先核查标注了 ⚠️ 的条目（AI可信度较低）。
```

## Part 3：关键流程复核

### 执行方法

从 `03_系统架构.md` 和 `08_数据流与控制流.md` 中提取 5 个关键流程，每个流程以步骤列表呈现。

**必须包含的流程（根据项目类型选择）**：

| 流程 | 来源文档 | 说明 |
|------|---------|------|
| 系统上电启动流程 | `03_系统架构.md` | 必选 |
| 主要业务功能触发流程 | `04_功能模块.md` | 必选，选项目最核心的功能 |
| 通信数据收发流程 | `05_通信协议.md` + `08_数据流.md` | 有通信模块时必选 |
| 参数读写/持久化流程 | `06_关键参数表.md` | 有EEPROM/Flash存储时必选 |
| 异常/报警/故障处理流程 | `07_已知问题.md` | 尽量包含 |

### 每个流程格式

```markdown
### 流程{N}：{流程名称}

**关键文件**：`{主文件}` + `{相关文件}`

**步骤**：
1. {步骤1} — 来源：`{文件}:{行号}`
2. {步骤2} — 来源：`{文件}:{行号}`
3. {步骤3} — 来源：`{文件}:{行号}`
...

**客户核实**：
☐ 步骤顺序和描述基本正确
☐ 有步骤描述错误（请备注：___________）
☐ 有步骤缺失（请备注：___________）
```

## Part 4：盲测问题集

### 执行方法

从 `11_常见问题清单.md` 和 `12_历史经验库.md`（如果存在）中挑选 20 道有代表性的问题，不附答案，让客户直接向 AI 问答系统提问并打分。

**选题原则**：
- 覆盖配置类、调试类、功能类各 5-7 题
- 优先选"客户最可能真实提问"的题
- 不选过于简单（单行能答）或过于模糊（需要上下文）的题

### 每题格式

```markdown
**B{序号}** [{类别}] {问题}

_（直接把这个问题发给 AI 问答系统，记录回答，然后评分）_

客户评分：☐ 答对了 ☐ 部分正确 ☐ 答错了 ☐ 没有回答
备注：___________
```

### Part 4 末尾汇总

```markdown
## 盲测得分汇总

| 结果 | 题数 | 占比 |
|------|------|------|
| ✅ 答对 | X | X% |
| ⚠️ 部分正确 | X | X% |
| ❌ 答错 | X | X% |
| 🚫 未回答 | X | X% |

**盲测通过标准**：答对率 ≥ 75%（含部分正确按0.5分计）

**实际得分**：{X}分 / 20分 → {✅通过 / ❌未通过}
```


## 验收包的交付说明（给服务方的提示）

生成 `验收包.md` 后，发送给客户时附上以下说明：

> 您好，项目知识库已完成初步构建，请用附件中的验收包核实知识库质量。
>
> **建议先看 Part 1**，20道题大概30分钟，能快速判断整体质量。
> 如果发现明显错误，请直接在文档里备注，我会逐条更新。
>
> 验收通过标准：Part 1 答对 ≥ 18题，Part 4 盲测得分 ≥ 75%。
