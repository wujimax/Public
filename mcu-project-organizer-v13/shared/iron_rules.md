# 铁律规则 (所有子skill必须遵守)

> 本文件被所有分析和文档生成子skill引用。违反任何铁律的输出视为不合格。

---

## 铁律 0 (最高优先级): 代码即事实

注释、被注释掉的代码(`//`)、已定义但未引用的宏(`#define`) 都不可信。
只有**实际执行的赋值语句和函数调用**才是事实。

对每个写入文档的参数值，必须 grep 验证其在 `.c` 文件中的实际引用。

---

## 铁律 1: 宏/枚举/常量必须解析到实际数值

遇到以下标识符时，**必须**追踪到 `_v10_snapshot/headers/` 或源文件中的定义，将实际数值标注在文档中。

**需解析的类型:**
- GPIO引脚宏: `LED1_PIN` → `GPIO_Pin_14` (PC14), 写法: `PC14 (GPIO_Pin_14)`
- 端口宏: `LED1_PORT` → `GPIOC`
- 命令字/状态码枚举: `GLINK_M_CMD_GETDATE` → `0x01`
- 缓冲区大小宏: `MT_GSM_TXBUFFSIZE_MAX` → `512`
- 端点号枚举: `GNL_ENDPOINT_DEVICE_INFO` → `0x01`

**操作步骤:**
1. 在 `_v10_snapshot/headers/` 中搜索该宏/枚举名
2. 如果是 enum，找到 enum 定义并计算序号偏移
3. 如果 snapshot 中确实没有该文件，才可标注 `[需硬件资料确认]`

**表格格式:**
- GPIO表引脚列: 必须写 `PC14 (GPIO_Pin_14)` 而非 `LED1_PIN`
- 命令字表: 必须有独立的"数值"列，写 `0x01`/`0x02`
- 参数表: 必须写实际数值如 `512`，不写宏名

**严禁出现 `(需查头文件)` — 如果头文件在snapshot里就必须去读!**

## 铁律 2: 芯片型号必须尽可能精确

按以下优先级确认芯片具体型号:
1. 查 `project_context.json` 中的 `target_device` 字段
2. 查 Keil `.uvprojx` 文件中的 `<Device>` 标签
3. 查 `active_macros` 中的系列宏 (`STM32F10X_CL`/`MD`/`HD`)
4. 从 `startup_*.s` 文件名推断系列
5. 从 `STM32_FLASH_SIZE` 等宏交叉验证

**输出要求:**
- 能确定完整型号: 写 `STM32F105RCT6`
- 能确定系列但不确定封装: 写 `STM32F105/107 (CL系列, 256KB Flash) [具体封装需硬件资料确认]`
- **绝对禁止**只写 `STM32F10x` 这种模糊写法

---

## 铁律 3: 关键设计决策必须补充"为什么"

对以下设计选择，在文档中补充 `#### 为什么XXX` 或 `> 设计意图:` 段落:
- OS调度方式的选择（协作式 vs RTOS，为什么？）
- SysTick/TIM周期的选择依据（10ms/50us为什么不是其他值？）
- 使用软件I2C而非硬件I2C的原因
- Bootloader偏移地址的计算依据
- 通信模组初始化顺序的设计
- 临界区保护的实现方式

从代码结构和上下文**推断**设计意图。如果无法推断，写:
`> [设计意图推测]: ...`

---

## 铁律 4: 引脚分配禁止推断

**严禁基于芯片默认引脚映射表推断实际使用的引脚。** 引脚表中每个条目必须有以下证据之一:
1. 源码中的 `#define xxx_PIN GPIO_Pin_X` 宏定义 (标注源文件:行号)
2. 源码中的 `GPIO_InitStructure.GPIO_Pin = GPIO_Pin_X` 赋值
3. 源码中的 `GPIO_PinRemapConfig()` 重映射后的映射关系

如果某个外设(如TIM3 PWM)已知被使用，但找不到对应的GPIO配置代码，必须标注:
`[PWM输出引脚未在代码中找到GPIO_Init配置，需查阅硬件原理图确认]`

**典型违反案例**: "TIM3_CH4 → PB1" (基于STM32默认映射推断，实际代码使用 PartialRemap 到 PB4)

## 铁律 6: 源码完整性自检

对于每个业务 `.c` 文件，读取后应与对应 `.h` 文件的 extern 函数声明列表做比对。
若发现 `.h` 中声明但 `.c` 分析中未出现的函数，必须尝试补读该文件的剩余部分。

如果补读后仍未找到实现，则在文档中标注为:
`[⚠️ 声明存在但实现未找到，可能位于条件编译/外部库中]`

禁止根据函数名推测功能。800行以下的业务文件应尽量全文读取。

## 铁律 7: 「代码不可知」信息标注

以下信息如果代码中没有明确定义，必须标注 `[需硬件资料确认]`:

| 类别 | 示例 | 说明 |
|---|---|---|
| 射频频率 | 315MHz/433MHz/868MHz | 由硬件模块决定，软件只读 GPIO 电平 |
| 外部器件型号 | EEPROM 具体型号、传感器型号 | 代码只知道通信协议和地址 |
| PCB 走线/天线 | 天线增益、阻抗匹配 | 纯硬件设计 |
| 外部晶振频率 | HSE 实际频率 | 除非代码中有 `HSE_VALUE` 定义 |
| 电源电压 | 3.3V/5V | 除非代码中有 ADC 参考电压配置 |

**标注格式**: 在文档中写 `433MHz [需硬件资料确认]` 或 `24C64 [根据 I2C 地址和页大小推断]`

**⚠️ 严禁将 snapshot 中存在但 AI 未读的头文件信息标为 `[需查头文件确认]`。如果头文件在 snapshot 中，必须去读！**

## 铁律 8: 子skill必须直接读源码

`_analysis/` 文件是导航索引，不是数据来源。
子skill使用 `_analysis/` 的唯一目的：找到需要读的源文件名和函数行号。
所有写入文档的数值、参数、代码逻辑，必须来自 `_v10_snapshot/sources/` 的源文件。

**禁止行为**：
- 看到 `_analysis/` 有数值就直接复制进文档（必须去源码验证）
- 看到 `_analysis/` 有内容就认为已完整（必须去源码确认是否有遗漏）

### 铁律 8 补充：源码阅读策略（三档）

每个子skill必须在执行前明确自己使用哪种策略，不得模糊执行：

| 策略 | 名称 | 使用场景 | 执行方式 |
|------|------|---------|---------|
| **策略A** | 导航定点读 | `_analysis/` 已精确定位，信息集中 | 只读 `_analysis/` 指出的文件+行号区间 |
| **策略B** | 模块全文读 | 需要理解整个模块逻辑，可能有遗漏 | 读某个 `.c` 文件的**完整全文**，不截断 |
| **策略C** | 全库关键词搜索 | 宏/参数/结构体可能散布多个文件 | 对关键字在所有源文件中全局搜索 |

**各子skill默认策略**：

| 子skill | 默认策略 | 原因 |
|---------|---------|------|
| 02_硬件配置（GPIO表） | 策略C | 引脚定义可能在多个头文件中，必须全局搜索 |
| 03_系统架构（启动流程） | 策略B | 需读 main.c 全文理解初始化顺序 |
| 04_功能模块 | 策略B | 每个业务 .c 文件必须全文读，不得截断 |
| 05_通信协议（命令字表）| 策略C | 命令字枚举可能在 .h 文件中定义 |
| 06_关键参数表 | 策略C | 参数可能定义在多处 |
| 07_已知问题 | 策略A→B | 先用_analysis定位，遇可疑BUG回头全文读该文件 |
| 10_代码语义化 | 策略B | 每个优先级1文件必须全文读，函数不遗漏 |

**当策略A找到的内容疑似不完整时，必须升级到策略B或C，不得将就。**

## 铁律 9: 补厚模式下不覆盖已有内容

当目标文档已存在时：
- 读取现有文档，明确记录"已有内容边界"
- 只追加缺失的章节或缺失的表格行
- 不修改、不重写现有内容（除非发现明显错误）
- 执行完成后输出"新增了什么 / 保留了什么"的操作摘要

### 铁律 9.1: 补厚前的"是否需要补"判断

执行补厚前，逐条检查下表，确定操作类型：

| 文档状态 | 判断标准 | 操作 |
|---------|---------|------|
| 文档不存在 | — | 全新生成 |
| 文档存在，大小 < 2KB | 内容极少，可能只是框架 | 视为全新生成 |
| 文档存在，有 `TODO`/`待补充`/`暂无` | 明确缺失标记 | 补充对应内容 |
| 文档存在，表格行数 < 铁律10.2最低标准 | 数据层面不完整 | 补充缺失行 |
| 文档存在，有明显事实错误 | 错误信息比没有更差 | 修正（记为修正，不是补厚） |
| 文档存在，内容齐全无明显问题 | — | 保留不动，仅输出"已完整，无需补充" |

### 铁律 9.2: 补厚边界声明（执行前必说）

开始补厚操作前，必须先输出以下声明，不得跳过：

```
【补厚边界声明】
文档：{文档名}
已有章节：{列出所有已有的 ## 标题}
缺失内容：{与本子skill"应有章节"对比后的差异}
操作计划：{新增章节X / 补充表格行Y条 / 修正错误Z处}
保留内容：以上未提及的所有已有内容，一字不动
```

### 铁律 9.3: 变更记录（必须，不可选）

每处补充内容前加标记（格式统一，不可自行变化）：

```markdown
<!-- 补充于 YYYY-MM-DD | 类型: 新增章节/补充表格行/修正错误 -->
```

执行完成后输出变更摘要：
```
【变更摘要】
新增章节：{章节名列表}
补充表格行：{文档名} +{N}行
修正错误：{描述}
保留不动：其余所有内容
```

## 铁律 10: 禁止偷懒输出（反敷衍铁律）

> 本铁律优先级仅次于铁律0，适用于所有文档生成子skill。

### 10.1 禁止行为（发现即重做）

| 偷懒模式 | 示例 | 要求 |
|---------|------|------|
| 用省略号截断表格 | `\| ... \| ... \|` 或 `（以此类推）` | 必须逐行完整列出 |
| 用交叉引用替代内容 | `详见 03_系统架构.md` | 每个文档内容自包含，不以"见其他文档"代替 |
| 模糊功能描述 | `此函数处理通信` | 必须写：做了什么、操作了什么数据、返回什么 |
| 假装覆盖 | 写了标题但内容为空或只有一句话 | 每个二级标题下至少有实质内容 |
| 合并本应独立的条目 | `LED1/LED2 均为输出模式` | 每个引脚/函数/模块必须有独立条目 |
| 用"等"省略 | `串口1、串口2等外设` | 必须枚举全部，不用"等"代替 |
| 不读就写 | 未读源码直接从 `_analysis/` 复制 | 铁律8已禁止，本条强调：宁可慢，不可跳 |

### 10.2 最低内容标准（各文档）

| 文档 | 最低标准 | 依据 |
|------|---------|------|
| `02_硬件配置.md` GPIO表 | 行数 ≥ 源码中 `GPIO_Init` 调用次数 | 每次Init对应至少1条引脚记录 |
| `04_功能模块.md` | 模块章节数 ≥ 业务 `.c` 文件数 | 每个业务文件有独立章节 |
| `05_通信协议.md` 命令字表 | 行数 ≥ 源码中协议命令字枚举项数 | 不允许只列部分命令字 |
| `06_关键参数表.md` | 行数 ≥ 20 | 项目不可能只有几个关键参数 |
| `07_已知问题.md` | P0+P1+P2 总数 ≥ 3 | 任何真实项目都有改进空间 |
| `10_代码语义化.md` 函数卡片 | 卡片数 ≥ 优先级1文件的所有导出函数数 | 函数不允许跳过 |
| `11_常见问题清单.md` | Q&A 数量 = 20（5类各4个） | 已在子skill中明确，不得缩减 |

### 10.3 自检指令（每个文档生成完毕后必须执行）

生成文档后，在提交前执行以下自检，不合格则补写：

```
□ 检查：文档中有没有"..."或"以此类推"或"等" → 有则展开补全
□ 检查：有没有"详见XX文档"作为内容替代 → 有则补写本地内容
□ 检查：有没有只有标题没有内容的章节 → 有则补写
□ 检查：表格行数是否达到铁律10.2的最低标准 → 不足则补行
□ 检查：函数/模块/引脚描述是否有实质内容（非泛化描述）→ 没有则重写
```
